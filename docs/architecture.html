Architecture report

expjobserver

A small cluster manager: it schedules experiment commands onto a pool of reserved test machines over SSH and copies their results back. Two Rust binaries (expjobserver, j) plus one shell runner wrapper.

The three pieces

expjobserver (server)

Always-on. Holds the machine pool, the job queue, and a per-job state machine. Drives jobs forward on a background work thread, then rsyncs results back.

src/bin/server/

j (client)

Stateless CLI. Each invocation builds one protobuf request, sends it over TCP to 127.0.0.1:3030, prints the reply.

src/bin/client/

remote wrapper (RUNNER)

The deep module of this fork. The server execs it per job; it reads the first command token as the host and SSHes in to run the rest. Swapping this swaps where jobs run.

expjobserver_remote_wrapper.sh

How a job flows

From j job add to results on disk. The client never touches the machine — everything goes through the server and its runner.

sequenceDiagram

    participant J as j (client)
    participant S as expjobserver
    participant W as work thread / SM
    participant R as RUNNER (wrapper)
    participant M as remote machine
    participant C as copier
    J->>S: AddJobRequest{class, cmd, cp_results}  (protobuf/TCP)
    S-->>J: JobIdResp{jid}
    Note over S,W: job sits Waiting until a machine in its class is free
    W->>W: substitute {VAR} and {MACHINE}
    W->>R: RUNNER --print_results_path {MACHINE} cmd args...
    R->>M: ssh host  (scp the script first if it is local)
    M-->>R: stdout, incl. "RESULTS: "
    R-->>W: exit 0 + captured stdout
    W->>C: hand off  for copy
    C->>M: rsync results -> cp_results dir
    C-->>S: Done (with results)
The happy path in one line: a job is a command string scheduled to a class; the server fills in {VAR}/{MACHINE}, runs it on a free machine via SSH, and copies back whatever the job announces with RESULTS:.

Job lifecycle (the state machine)

Every job and setup task is one big state machine in src/bin/server/main.rs (TaskState), advanced by sm.rs. The work thread re-drives every non-terminal task each tick.

stateDiagram-v2
    [*] --> Waiting
    Waiting --> Held: j job hold
    Held --> Waiting: j job unhold
    Waiting --> Running: free machine in class
    Running --> CheckingResults: exit 0
    Running --> Error: non-zero exit / timeout
    CheckingResults --> CopyingResults: RESULTS line found
    CheckingResults --> Finalize: no results
    CopyingResults --> Finalize
    Finalize --> Done
    Finalize --> DoneWithResults
    Error --> ErrorDone
    ErrorDone --> Waiting: repeat_on_fail → clone
    ErrorDone --> [*]: failures exhausted
    Waiting --> Canceled: j job cancel
    Running --> Canceled
    Canceled --> Killed
    Done --> [*]
    DoneWithResults --> [*]
    Killed --> [*]

Setup tasks share this machine; on success they additionally register their machine into a class. Repeated failures of a machine move it out of its class (MachineStatus.failures).

Anatomy of a job command

The single most important thing to understand about this fork. A real line from scripts/hemem_baseline.sh:

j job add
foo
class
"{MACHINE} ~/workloads/run.sh -b graph500 -o results/baseline -r 3"
command string (one quoted arg)
./hemem_baseline_bwmon
results dir
▼ server substitutes & tokenises, then execs the runner ▼
{MACHINE}
→ host, consumed by wrapper
~/workloads/run.sh -b graph500 -o results/baseline -r 3
→ run on the remote over SSH

Gotcha 1 — {MACHINE} must come first.

The server passes only command tokens to the runner; the wrapper treats token #1 as the SSH host. Forget it and the wrapper tries to SSH to ~/workloads/run.sh.

Gotcha 2 — no shell operators.

The command is split_whitespace()'d and replayed as "$@". &&, |, >, ; are literal args. Put multi-step logic inside the invoked script.

Variables, matrices, and "env"

Variables {VAR}

Server-side text substitution into the command, set with j var set. Resolved before SSH — not a remote env var.

Matrix

Cartesian product over variable lists generates many jobs from one template. BENCH=a,b INTERVAL=1,2 → 4 jobs. Dump results with j job matrix csv.

Real env vars

None natively. Bake them into the command: "{MACHINE} FOO=bar run.sh".

Provisioning a machine (a separate path)

scripts/add_machine.sh prepares a freshly reserved CloudLab host and registers it. It runs over plain SSH/rsync and does not go through the job server or the runner wrapper — only the final step touches j.

Source map

Server · src/bin/server/

  • main.rs — entry; Server, Task/TaskState, work thread
  • sm.rs — state machine; spawns the runner
  • request.rs — one handler per client request
  • copier.rs — rsync results back
  • snapshot.rs + state.proto — persist/restore state
  • notify.rs — Slack notifications

Client · src/bin/client/ & shared

  • cli.rs — clap command tree
  • main.rs — build requests, send, dispatch
  • pretty.rs / stat.rs — table output
  • ../../lib.rs — {VAR}/{MACHINE} subst, cartesian product, ts helpers
  • ../../protocol.proto — RPC messages (built by build.rs)
  • expjobserver_remote_wrapper.sh — the RUNNER

Config seam: the wrapper sources example_config.sh (tracked defaults) then config.local.sh (gitignored real values), resolved relative to the wrapper so the server's working directory doesn't matter.
